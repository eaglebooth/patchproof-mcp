## Goal

Build a production-ready PatchProof MCP server (Cysic Arena Sprint) that audits software supply-chain risks across the full chain `dependency → vulnerability → reachability → remediation → verification → evidence`, exposed via 8 MCP tools over stdio and Streamable HTTP, with a polished browser demo and verifiable repository evidence.

- Constraints: Node.js 20, strict TypeScript, official MCP TypeScript SDK, Zod for input validation, npm-only initial scope (package.json + package-lock.json), CycloneDX SBOM, OSV isolated behind a typed adapter, deterministic mock for tests/demo.
- Security: no plaintext secrets in any output, no path traversal, no unsafe symlinks, no arbitrary shell, allowlisted verification commands only, no source file egress, no env/secret exposure.
- Quality bar: 8 MCP tools fully implemented + tested, ≥85% line coverage, transport smoke tests, Docker build, CI green, browser demo with bundled fixtures, acceptance evidence mapped 1:1 to AC.
- Architecture direction (per request): `src/{server,tools,scanners,parsers,sbom,osv,reachability,remediation,verification,reporting,security,transport,schemas,types,utils}` keeping business logic independent from MCP transport.
- Repo evidence required: README, plan.md, docs/{architecture,security-model,limitations,benchmarks,acceptance-evidence}.md, CONTRIBUTING, SECURITY, MIT, .env.example, Dockerfile, GH Actions workflow, 3 runnable MCP agent workflow examples, demo fixtures, deployment config.

## Assumptions

- Implementer is authorized to create, read, and run shell commands within the project workspace; no external network access is required for tests (mock OSV is sufficient and required by spec).
- npm 10+ is available on the runner for typecheck/test/build, and Docker is available for the container build step in CI.
- "Deterministic mock OSV" means a typed adapter that returns stable, fixture-based responses keyed by lockfile hash or dependency version, with no `Date.now()`/randomness in the result payload.
- The Streamable HTTP transport will bind to `127.0.0.1` by default with a configurable port; the deployable container will bind `0.0.0.0` only when `PATCHPROOF_HTTP_PUBLIC=1` is set.
- Browser demo will run as a static-served HTML/JS bundle inside the container (no DB) and call a thin Node-side demo API on `127.0.0.1` that wires to the same tool implementations over in-process calls, never accepting arbitrary filesystem paths from the browser.
- "Coverage shortfall" is acceptable and will be documented if and only if specific lines are platform-only or unreachable per Istanbul `/* c8 ignore next */` markers.
- This plan does not require user clarification before implementation; all blocking choices are resolved below.

## Acceptance Criteria

- AC-1: Repository scaffold matches the requested directory tree and uses Node.js 20, strict TypeScript (`"strict": true`, `noUncheckedIndexedAccess`, `exactOptionalPropertyTypes`), ESM modules, and the official `@modelcontextprotocol/sdk`.
- AC-2: All 8 MCP tools are registered and callable: `scan_repository`, `generate_sbom`, `audit_dependencies`, `detect_secrets`, `analyze_reachability`, `plan_remediation`, `verify_remediation`, `generate_evidence_report`.
- AC-3: Each tool has a Zod input schema; invalid input is rejected with a typed, redacted error and never reaches business logic.
- AC-4: `parse-manifest` and `parse-lockfile` use `jsonc-parser` (or `@slicknode/package-json`) and `@npmcli/lockfile-type` style structured parsing — no regex-based dependency extraction.
- AC-5: `generate_sbom` emits a CycloneDX 1.5 JSON object validated against the official CycloneDX JS schema in tests; components include name, version, purl, and licenses when available.
- AC-6: `OSVClient` is a typed interface with `LiveOSVClient` and `MockOSVClient`; live mode has timeout (default 5s), bounded retry (max 2, exponential backoff with jitter), in-memory TTL cache (default 1h), and per-minute rate limit (default 60).
- AC-7: `detect_secrets` returns only redacted fingerprints, file paths, line ranges, rule IDs, categories, and confidence; plaintext is never present in any return path, log, or test snapshot.
- AC-8: `analyze_reachability` classifies results as `confirmed | possible | unknown` with file:line evidence; "confirmed" requires an import/require matching the vulnerable package's resolved path.
- AC-9: `plan_remediation` never writes to disk; output is a ranked list of `{package, currentVersion, recommendedVersion, breakingChangeRisk: low|medium|high, affectedFiles[], verificationCommands[]}`.
- AC-10: `verify_remediation` accepts only allowlisted operations: `npm test`, `npm run lint`, `npm run build`, `npm audit --json`; rejects redirections, pipes, `;`, `&&`, `||`, `$()`, backticks, absolute paths outside the repo, and symlink escapes. Returns `{exitCode, durationMs, stdout, stderr}` with stdout/stderr redacted and size-capped (1 MiB each).
- AC-11: `generate_evidence_report` produces both a JSON report (with `schemaVersion`, `generatedAt`, `inputs`, `findings`, `remediation`, `verification`, `limitations`, `redactions`) and a self-contained HTML report (no external assets, inline CSS/JS) with a stable layout and accessible markup.
- AC-12: Path safety: every filesystem entry point resolves through `security/paths.ts` which uses `path.resolve` + `fs.realpath` and rejects any path whose realpath escapes the authorized repo root or contains unsafe symlinks. Directories in the default ignore set (`node_modules`, `dist`, `build`, `coverage`, `.git`, `.next`, `.cache`, `out`) are skipped.
- AC-13: Resource limits: scan caps at 50k files, 500 MiB total read, recursion depth 10, wall-clock 60s — enforced in a single `ResourceGovernor` and exceeded yields a typed `ResourceLimitError` with a redacted message.
- AC-14: stdio transport smoke test spawns the built server with `--transport=stdio` and a tiny in-process client that lists tools and calls each; all return without unhandled errors.
- AC-15: Streamable HTTP transport smoke test boots the server on an ephemeral port, hits `/mcp` with a JSON-RPC initialize + tools/list + one tool call, and shuts down cleanly.
- AC-16: `npm run lint`, `npm run typecheck`, `npm test`, `npm run coverage`, `npm run build` all exit 0 in CI. Coverage is ≥85% lines/branches/functions on the `src/` graph or the exact shortfall is enumerated in `docs/limitations.md`.
- AC-17: GitHub Actions workflow `.github/workflows/ci.yml` runs install, format check, lint, typecheck, test, coverage, build, and Docker build on push/PR to `main`.
- AC-18: Dockerfile produces a multi-stage image (`node:20-alpine` builder → `node:20-alpine` runtime as non-root `patchproof`) under 200 MiB; `docker build .` and `docker run patchproof --transport=stdio` succeed.
- AC-19: Browser demo (`web/`) is a static SPA that ships with bundled fixtures: `safe`, `vulnerable`, `secret-leak`, `malformed-lockfile`, `missing-lockfile`, `unavailable-osv`. Selecting each calls the demo API (which routes to MCP tools) and renders findings, reachability, remediation, verification, and report preview. Demo state is clearly labeled `FIXTURE / DEMO MODE` in the UI; arbitrary server paths are never accepted.
- AC-20: Repo evidence: `README.md`, `plan.md`, `docs/architecture.md`, `docs/security-model.md`, `docs/limitations.md`, `docs/benchmarks.md`, `docs/acceptance-evidence.md`, `CONTRIBUTING.md`, `SECURITY.md`, `MIT LICENSE`, `.env.example`, `Dockerfile`, `.github/workflows/ci.yml`, `examples/agent-workflows/{github-copilot,claude-code,codex}.md` all exist and are non-empty. Each README claim must be backed by an actual file/test/command.
- AC-21: Secret-leakage audit: a CI step greps JSON reports, HTML reports, dist output, and snapshots for known plaintext test secrets (e.g., `AKIAIOSFODNN7EXAMPLE`, `ghp_xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx`, `xoxb-…`) and the test fixtures themselves. Test fails on any hit.
- AC-22: `docs/acceptance-evidence.md` contains a 1:1 table mapping each AC above to: the implementing file(s), the test file(s), and a one-line reproducible command.
- AC-23: Final summary in `docs/acceptance-evidence.md` records implemented features, passing commands, coverage %, Docker status, known limitations, and any unmet AC.

### Key technical decisions

- Use the **official MCP TypeScript SDK** (`@modelcontextprotocol/sdk`); tool registration via `server.tool(name, schemaDescription, zodShape, handler)`.
- **No regex for dependency parsing.** Manifest parsed as JSON; lockfile parsed by lockfile-version branching (`"lockfileVersion": 1` uses npm v1 shape with `dependencies` nested; v2/v3 use `packages` keyed by `node_modules/<name>`).
- **Determinism**: mock OSV keys responses by `sha1(stableStringify(dependencies))`; all reports embed `schemaVersion` and `generatedAt` derived from an injected `Clock` to make tests reproducible.
- **Secret redaction is layered**: (1) raw secret strings never enter any return value — only fingerprints; (2) redaction pass on paths, URLs, and free-form error messages; (3) CI grep audit.
- **Resource limits are single-source**: `ResourceGovernor` constructed per request from config; tools call `governor.checkFile()`, `governor.checkDir()`, `governor.checkTime()`.
- **No source egress**: the only outbound HTTP is OSV, and it sends only `{package:{name,ecosystem}, version}`. Source content is never serialized to the network layer.
- **Browser demo isolation**: the demo API accepts only `fixtureId` from a static allowlist; even if a user crafts a request body with `path`, the handler ignores it and returns 400.
- **Coverage**: Vitest with `@vitest/coverage-v8`; threshold enforced in `vitest.config.ts` (`thresholds: { lines: 85, branches: 80, functions: 85, statements: 85 }`). If below, `scripts/coverage-report.mjs` writes the exact shortfall into `docs/limitations.md`.

### Risk controls

- **Path traversal** — every read goes through `security/paths.ts::safeResolve(root, candidate)`; symlink escape test fixture is part of the suite.
- **Command injection** — `verification/runner.ts` uses `child_process.spawn` with `shell: false`; arg validator runs before spawn; allowlist is the only source of accepted commands.
- **Secret leakage** — `scripts/audit-secrets.sh` greps for canonical example strings; tests assert redaction in snapshots; no `console.log` in handlers (logger redacts by default).
- **Transport regressions** — smoke tests spawn the built binary, not the dev entry, so they catch packaging issues.
- **Untested coverage** — `/* c8 ignore next */` only with a justification comment; uncovered code blocks are reported in `docs/limitations.md` with file:line.

## Out of Scope

- Non-npm ecosystems (PyPI, Maven, Go modules, Cargo) — extensible later but not in this sprint.
- Auto-installing dependency upgrades (the `plan_remediation` tool is read-only by design).
- Uploading findings to remote dashboards/SIEMs.
- A production multi-tenant deployment topology (the docker image is single-tenant local-first; multi-tenant authn/authz is deferred).
- License-policy enforcement beyond surfacing declared licenses in the SBOM.
- SBIF/SARIF export formats (only CycloneDX JSON + self-contained HTML are produced).
- Any browser-side path or filesystem input (the demo strictly uses bundled fixtures).
